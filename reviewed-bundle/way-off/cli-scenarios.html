Scenario 01
The Most Common Nightmare
The Budget Blowout
An engineer asks the AI agent to refactor the auth module. Without governance, the agent retries on failure, burning tokens and money with no ceiling. With MartinLoop, the hard cap kicks in — the agent stops, rolls back, and produces a receipt.
Without MartinLoop (Ralph)
ralph — no governance
$ ralph run "refactor auth module"
💸

With MartinLoop

Scenario 02
The Sneakiest Problem
The Silent Failure
The agent says the task is done. It lied. Tests are still failing. Without a verifier, the engineer ships broken code. MartinLoop grades the output from actual test results — not from what the AI claims.
Ralphy — claims success
ralphy — no verifier
$ ralphy run "add rate limiting to API" ⟳ Running... ✓ Task completed ✓ Files modified: src/api/rate-limit.ts ✓ Model reports: implementation correct Status: SUCCESS ← model's own report Tests run: Not verified Cost: $2.40 No verifier. [ 3 hours later: tests fail in CI ] [ Team reverts. Day lost. ]
🚨
Hallucinated completion
The AI said it worked. It didn't. No verifier caught it. This is how bugs ship.
MartinLoop — verifier catches it
martin — verified
$ martin run "add rate limiting to API" \ --verify "pnpm test" --budget 2.00 ✓ Leash armed ✓ Budget cap $2.00 ⟳ Attempt 1... ✓ Files modified: src/api/rate-limit.ts ⟳ Running verifier: pnpm test... ✗ 2 tests failing: rate-limit.test.ts Grade: C — incomplete implementation ──────────────────────────────── ⚠ Grade C: requires review before merge ✓ Rollback available — no bad code shipped ✓ Signed audit: shows exactly what failed Engineer notified with proof bundle.
⚠️
Grade C — flagged before merge
Tests ran. 2 failures caught. Engineer reviews with full evidence. No broken code in main.

"Done"

The AI cannot grade its own work. MartinLoop runs your actual test command and derives A/B/C from the real result — not from model confidence. Our 8/10 verified completion (vs 10/10 claimed by Ralph/Ralphy) is an honest number that builds trust.
