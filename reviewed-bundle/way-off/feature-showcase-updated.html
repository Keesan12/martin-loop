Core Runtime
Phases 1–10 · The governed loop engine
Ph 1–2
Governed Loop Engine
State machine that wraps any AI coding agent: GATHER → ADMIT → PATCH → VERIFY → RECOVER → ESCALATE → ABORT → HANDOFF. Every transition is explicit and logged.
Why it matters: Unguarded loops end when the model decides they're done. Martin ends when the work is provably complete or the budget runs out — whichever comes first.
✓ Approved
Ph 3
Hard Budget Kill
Hard USD cap enforced at runtime. When actual spend reaches the cap, the loop is killed immediately — not warned, not softly throttled. Zero overage.
Why it matters: Every other runner gives you a soft warning. Martin gives you a hard stop. The difference is $0 vs $47 surprise charges at 3am.
✓ Approved · Hard cap, not hints
Ph 4
Filesystem Rollback
Every failed or rejected attempt triggers a full filesystem rollback. Files written during the attempt are removed. The workspace is left exactly as it was before the attempt started.
Why it matters: A failed AI run that leaves half-written files is worse than a failed run that leaves nothing. Rollback is the safety net that makes overnight runs trustworthy.
✓ Approved
Ph 5
Adapter Layer (Claude / Codex / Any)
Pluggable adapter interface. Ships with Claude CLI and Codex CLI adapters. Any provider with a CLI or API can be wrapped. Adapters report cost, model, and output — the runtime governs the rest.
Why it matters: Model-agnostic from day one. Switch providers without rewriting governance logic.
✓ Approved
Ph 6
Resumable Loops (martin resume)
Checkpoints written before each attempt. If the process crashes or the machine restarts, `martin resume <loopId>` picks up from the last committed checkpoint — no work lost.
Why it matters: Overnight runs on flaky hardware or preemptible cloud instances can resume without paying to restart from scratch.
✓ Approved
Ph 7
Policy-as-Code (martin.policy.yaml)
Layered config system: built-in defaults → ~/.martin/policy.yaml → repo root → CLI flags. Controls budget caps, blocked commands, max attempts, CVE severity thresholds, SIEM endpoint, and more.
Why it matters: Enterprise teams need governance that follows the code — not per-user settings that drift. Policy-as-code means reviewable, diffable, auditable AI controls.
✓ Approved
Safety & Governance
Phases 8–15 · The verification and audit layer
Ph 8
14-Class Verification Leash
Before any shell command executes, it passes through a 14-class regex trap. Classes include: pipe-to-bash, rm-rf, git-force-push, npm-publish, path-traversal, scope-escape, and 8 more. Block fires before execution — not after.
Why it matters: The only way to trust an AI coding loop is to verify its output before it touches anything. The leash is the enforcement point that makes all other guarantees possible.
✓ Approved · 14 classes, pre-exec
Ph 9
Ed25519-Signed Audit Trail
Every attempt writes a tamper-evident JSON artifact signed with Ed25519. Contains: grade, verifier pass/fail counts, cost, patched files, all leash decisions, and a replayable flag. Verifiable offline without Martin.
Why it matters: "The AI did it" is not an audit trail. A signed, replayable artifact that any engineer can verify independently is. Required for enterprise compliance.
✓ Approved
Ph 10
Verifier-Graded Patch Acceptance
After each attempt, the user-supplied verify command runs. Pass/fail count determines a grade (A/B/C). Grade A/B: patch accepted. Grade C: patch rejected, rollback triggered, retry within budget.
Why it matters: The verifier decision comes from actual test output — not model confidence. A loop that accepts patches based on what the model thinks is done is not governed.
✓ Approved · 80% verifier pass rate
Ph 11
CVE-Watch Dependency Scanning
Pre-run and post-patch CVE scan of dependencies. Configurable block threshold (CRITICAL/HIGH/MEDIUM/LOW). failClosed option prevents the loop from proceeding if the scanner is unavailable.
Why it matters: AI agents that install packages without CVE checks introduce supply-chain risk at machine speed. Martin blocks at install time.
✓ Approved
Ph 12
SLSA Build Level 3 Provenance
Published artifacts carry SLSA Build Level 3 provenance attestation. Build inputs, builder identity, and output digests are recorded and verifiable via sigstore/cosign.
Why it matters: Supply chain attacks target the build pipeline. SLSA L3 provenance gives downstream consumers cryptographic proof that the artifact was built from the declared source.
✓ Approved
Ph 13–14
63/63 Adversarial Gate Checks
63 declared adversarial scenarios, all passing. Covers: verifier leash bypass, grounding contradictions, scope-escape, path-traversal, budget-governor bypass. Zero false positives. Reproducible in CI.
Why it matters: A safety system you can't verify is a marketing claim. 63 replayable gate checks that any engineer can run is a verified safety record.
✓ Approved · 63/63, 0 false positives
Ph 15
SIEM Streaming (OCSF/CEF)
Runtime events stream to any SIEM in OCSF or CEF format. Each leash decision, budget event, verifier result, and rollback action is a discrete, structured event. Configuration via martin.policy.yaml.
Why it matters: Enterprise security teams need AI agent activity in their existing SIEM. Martin is the first AI coding runtime with native SIEM integration out of the box.
✓ Approved

Observability
Phases 23–25 · Full runtime visibility
Ph 23
OpenTelemetry Integration
9 named span types exported via OTel: budget decisions, leash evaluations, verifier runs, rollback events, grounding scans, proof registrations, gateway policy decisions, and router decisions (with cost_per_1k_tokens + trust_tier attributes). Compatible with any OTel-compliant backend.
Why it matters: You can't manage what you can't measure. OTel spans mean Martin runs appear in your existing Datadog, Honeycomb, or Grafana dashboards alongside your other services.
✓ Approved
Ph 24
Natural Language Explanations (martin explain)
`martin explain <loopId>` returns a human-readable summary of any run: what was attempted, what the verifier said, why the patch was accepted or rejected, what the cost breakdown was.
Why it matters: Engineering managers and product owners need to understand what the AI did without reading JSON artifacts. martin explain bridges that gap.
✓ Approved
Ph 25
Replayable Decision Log
Every accept/reject decision is replayable offline. Replay inputs → same output deterministically. A replay mismatch is treated as a P0 defect and blocks the next release.
Why it matters: For ML research and compliance: the system is auditable not just by reading logs, but by re-running the exact decision and getting the same answer.
✓ Approved
Intelligence Layer
Phases 26–31 · Memory, routing, graph context
Ph 26–28
Autonomy Envelope (12/12 safe tasks)
Scoped autonomy framework: 12 safe task scenarios where Martin operates without human confirmation — tests, docs, changelogs, vitest snapshots, benchmarks, and contracts refactors all covered. Formal escalation ledger tracks when the agent exceeded its declared scope.
Why it matters: "Autonomous" as a blanket claim is dangerous. An autonomy envelope with explicit safe/unsafe task boundaries is the responsible alternative.
✓ Approved
Ph 31
Memory Palace — Cross-Run Recall
Persistent memory per workspace/project. Recalls relevant prior runs by scoring: failure-class match (+30), intervention match (+14), token overlap (+6 each), completion memory (+4). Fail-open: memory errors never block a run.
Why it matters: Without memory, every run starts blind. With Memory Palace, Martin knows which files regressed last time, which fixes worked, and which approaches failed — before the first token is spent.
✓ Approved
Ph 31
Model Router — Cost-Aware Routing
Dynamically selects the optimal model/provider per attempt based on: budget pressure (healthy/soft_limit/hard_limit), failure class (verification_failure → high-trust), complexity score, remaining budget. Six routing heuristics in priority order.
Why it matters: Simple tasks don't need expensive models. Verification failures need high-trust models. The router automatically spends your budget where it matters most.
✓ Approved · 30–60% cost reduction on typical tasks
Ph 31
Graph Hotspots — Repository Risk Map
Reads a JSON adjacency graph of the codebase and identifies files with high regression counts. Files with regressionCount > 2 generate critical warnings injected into the model's context before patching begins.
Why it matters: The AI can't know which files have burned you before without being told. Graph Hotspots gives Martin institutional memory about which parts of the codebase are dangerous.
✓ Approved
MCP Gateway
Phase 32 · Governed external tool access
Ph 32
MCP Server Registry — Tool Policy Engine
Registers external MCP servers with trust tiers (system/verified/untrusted). Evaluates every tool call against allowlist, blocklist, dangerous-pattern detection, and execution profile (ci_safe blocks all untrusted servers). OTel span emitted per decision.
Why it matters: MCP tools are powerful. Ungovened MCP tool calls inside an AI loop are a serious attack surface. The Registry enforces policy before any external tool executes.
✓ Approved
Ph 32
MCP HTTP Transport — JSON-RPC 2.0
Dispatches governed JSON-RPC 2.0 calls to external MCP servers over HTTP POST. 30-second AbortController timeout per call. Vault headers injected per-call. Proper error classification (network, protocol, timeout, RPC error).
Why it matters: Reliable, observable MCP calls with timeout enforcement and vault-based auth. The transport layer is the bridge between Martin's governance and external tool ecosystems.
✓ Approved
Ph 32
Token Vault — Static & OAuth Proxy
StaticTokenVault: in-memory Map of pre-configured headers per server. OAuthProxyVault: stub for client-credentials flow (replace before production OAuth). Vault interface is the abstraction — swap implementations without touching transport code.
Why it matters: Credentials for external MCP servers need to be managed centrally, not hardcoded per-server. The vault pattern makes credential rotation a one-line config change.
