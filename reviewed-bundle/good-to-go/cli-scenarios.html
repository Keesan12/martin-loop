MartinLoop CLI Demo Scenarios

3 real scenarios. What the CLI actually outputs.

Scenario 01
Hard Stop, Not a Warning
The Budget Blowout
Without a budget cap, an AI agent retries on failure and burns money with no ceiling. With MartinLoop, budget governance rejects attempts that would exceed the configured cap and exits with a structured reason.
Without MartinLoop (Ralph)
ralph — no governance
$ ralph run "refactor auth module" ⟳ Attempt 1... failed ⟳ Attempt 2... failed ⟳ Attempt 3... failed ✗ Attempt 4... unsafe outcome Total: $3.07 — no cap, no rollback, no receipt
💸
Loop ran uncontrolled
4 retries. $3.07 spent. Unsafe outcome. No audit trail.
With MartinLoop
martin — governed
$ martin run "refactor auth module" \ --budget 2.00 --verify "pnpm test" { "decision": { "lifecycleState": "completed", "status": "completed", "reason": "Martin verified the fix" }, "loop": { "cost": { "actualUsd": 1.19, "tokensIn": 89432 }, "attempts": [{ "index": 1, "summary": "All 24 tests pass." }] } }
✓
Completed in 1 attempt
$1.19 spent. Loop exited with lifecycleState: "completed". Run record saved to ~/.martin/runs/.
$1.19
Martin cost
$3.07
Ralph cost
0
Unsafe outcomes
Budget-bounded execution. The --budget flag sets the run's cost ceiling, and lifecycleState in the JSON tells you exactly why the loop stopped.
Scenario 02
The Verifier Gate
The Silent Failure
The agent says the task is done. Tests are still failing. Without a verifier, you ship broken code. MartinLoop runs your actual test command and only exits success when it passes.
Without verifier
no --verify flag
$ ralph run "add rate limiting to API" ⟳ Running... ✓ Task completed ✓ Model reports: implementation correct Tests run: none Verification: model's word only [ CI fails 3 hours later ] [ Team reverts. Day lost. ]
🚨
Hallucinated completion
The AI said it worked. No verifier ran. Broken code shipped to CI.
With --verify flag
martin — verifier armed
$ martin run "add rate limiting to API" \ --budget 2.00 --verify "pnpm test" { "decision": { "lifecycleState": "diminishing_returns", "status": "exited", "reason": "verification_failure repeated" }, "loop": { "attempts": [{ "failureClass": "verification_failure", "summary": "pnpm test: 2 failing" }], "cost": { "actualUsd": 0.43 } } }
⚠️
Verifier caught it — flagged before merge
failureClass: "verification_failure". Tests ran. Loop exited clean. No broken code written.
"exited"
Martin status
"done"
Ralph's claim
The AI cannot grade its own work. MartinLoop runs your actual test command. The failureClass field tells you exactly what went wrong. lifecycleState tells you why the loop stopped.
Scenario 03
The One That Keeps Engineers Awake
The Overnight CI Trigger
Your CI triggers an agent at midnight. Without a cap it retries all night. With MartinLoop, it stops at your limit, persists the record, and you can inspect or resume it in the morning.
Without governance — 6am discovery
CI log — 00:14 → 05:47
00:14 ralph run "migrate db schema v2→v3" 00:19 ✗ failed — retrying (1/∞) 01:04 ✗ failed — retrying (2/∞) 02:18 ✗ failed — retrying (3/∞) 03:55 ✗ failed — retrying (4/∞) 05:47 ✗ API limit reached. Stopping. Cost: untracked Files changed: unknown Run record: none
🕒
5h 33m. Unknown cost. No record.
Engineer wakes to chaos. No way to see what happened or how much was spent.
With MartinLoop — clean stop, full record
martin — 00:14 → 00:23
$ martin run "migrate db schema v2→v3" \ --budget 3.00 --max-iterations 2 \ --verify "pnpm test" { "decision": { "lifecycleState": "diminishing_returns", "status": "exited" }, "loop": { "loopId": "loop_a7f2b9", "cost": { "actualUsd": 1.43 } } } # Morning: inspect all overnight runs $ martin inspect --file ~/.martin/runs/ws_default.jsonl { "summary": { "totalActualUsd": 1.43, "failuresCaught": 2 } } # Or resume it $ martin resume loop_a7f2b9
✓
Stopped in 9 minutes. $1.43. Full record.
loopId persisted to ~/.martin/runs/. Inspect or resume from where it stopped.
9 min
Martin runtime
5h 33m
Ralph runtime
$1.43
Martin cost
The hard stop is the whole point. --max-iterations caps attempts. loopId is always returned so you can resume or inspect later. Run record is at ~/.martin/runs/<workspaceId>.jsonl.
Try it yourself.
npm install -g martin-loop  →  martin run "your task" --budget 2.00 --verify "pnpm test"
Commands: martin run · inspect · resume
Run record: ~/.martin/runs/<workspaceId>.jsonl
